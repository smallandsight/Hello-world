# 古月租车小程序 - 软件框架设计文档

> **文档版本**：V1.0  
> **编制日期**：2026-04-05  
> **基于文档**：《需求分析文档》V3.0 + 《开发计划》V1.0  
> **架构模式**：模块化单体 + 清晰边界（DDD领域驱动设计）  
> **团队规模**：1-2人全栈团队

## 一、架构设计决策记录（ADR）

### ADR-001: 采用模块化单体架构

| 项目 | 内容 |
|------|------|
| **状态** | 已接受 |
| **上下文** | 1-2人团队、26周开发周期、需要快速交付MVP。微服务架构对小型团队来说复杂度过高，运维成本大。 |
| **决策** | 采用**模块化单体架构**，后端使用NestJS，通过模块(Module)划分业务边界，严格遵循DDD领域驱动设计原则。 |
| **后果** | 优点：开发效率高、部署简单、事务一致性易保证、学习成本低。<br>限制：需要严格的代码规范防止模块间耦合。<br>演进路径：未来业务增长可按限界上下文拆分为微服务。 |

### ADR-002: 前后端分离，独立部署

| 项目 | 内容 |
|------|------|
| **状态** | 已接受 |
| **上下文** | 用户端(支付宝小程序)与商家端(Web后台)属于不同用户群体，技术栈差异大。 |
| **决策** | 采用前后端分离架构：后端提供RESTful API，前端独立开发和部署。 |
| **后果** | 前后端可并行开发，便于分工协作。API契约成为协作桥梁，需要完善的接口文档。 |

```
│         │                                 │                                   │
│         ↓                                 ↓                                   │
│    ┌─────────┐                     ┌─────────────┐                            │
│    │ 已取消  │                     │  已完成      │                            │
│    │  (70)   │                     │    (60)      │                            │
│    └─────────┘                     └─────────────┘                            │
│                                                                               │
│    ┌─────────┐                                                                │
│    │  异常   │  ◄───── 任意状态 ────►  损坏/违章/事故等异常                      │
│    │  (80)   │                                                                │
│    └─────────┘                                                                │
│                                                                               │
└──────────────────────────────────────────────────────────────────────────────┘
```

**状态码定义：**

| 状态值 | 状态名 | 允许操作 | 下一状态 |
|-------|--------|---------|---------|
| 10 | 待支付 | 支付、取消 | 20/70 |
| 20 | 待取车 | 扫码取车、取消(需退款) | 30/70 |
| 30 | 使用中 | 续租、还车 | 40 |
| 40 | 待结算 | 结算确认 | 60 |
| 60 | 已完成 | 评价、申请发票 | 终态 |
| 70 | 已取消 | 无 | 终态 |
| 80 | 异常 | 异常处理 | 60/70 |

```typescript
interface JwtPayload {
  userId: number;          // 用户ID
  userType: 'user' | 'staff';  // 用户类型
  roles?: string[];       // 角色列表(商家端)
  permissions?: string[];  // 权限列表(商家端)
  storeId?: number;       // 所属门店ID(商家端)
  iat: number;            // 签发时间
  exp: number;            // 过期时间
}

// Token过期时间
const TOKEN_EXPIRE = {
  user: 7 * 24 * 60 * 60,    // 用户端7天

};
```
